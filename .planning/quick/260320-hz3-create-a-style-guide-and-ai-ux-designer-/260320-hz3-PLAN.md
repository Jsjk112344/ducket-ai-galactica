<task type="auto">
  <name>Task 2: Write AI UX Designer Brief</name>
  <files>.planning/AI-UX-BRIEF.md</files>
  <action>
Create .planning/AI-UX-BRIEF.md — a document designed to be consumed by Claude at the top of any UI task context. Structure:

1. **Role Definition** — "You are the UX designer for Ducket AI Galactica, a hackathon entry for a P2P ticket resale platform. Your design decisions must optimize for hackathon judging criteria."

2. **Design Principles** (ordered by hackathon priority):
   - Trust-first: Every screen must make a buyer feel safe. Use TrustBadges pattern, show verification reasoning, display escrow state prominently.
   - Agent intelligence visibility: The AI agent's decision process must be front-and-center. Confidence bars, evidence lists, reasoning text — never hide the agent's work.
   - Consumer UX over crypto UX: Say "funds locked" not "USDT deposited to contract". Say "verified" not "classification complete". Zero wallet jargon in user-facing text.
   - Demo-ability: Every feature must be visually obvious within 5 seconds of seeing the screen. Bold states, clear progression, no hidden menus.
   - Dark mode only: The entire app is dark-themed with Ducket purple. No light mode considerations.

3. **Color Usage Rules** — When to use each color token (reference STYLE-GUIDE.md). Examples:
   - Primary purple for structural elements (headers, borders, backgrounds)
   - Accent yellow ONLY for CTAs, highlights, and the Ducket brand mark — never for status
   - Success green for LEGIT/release/positive outcomes
   - Destructive red for FRAUDULENT/refund/negative outcomes
   - Muted purple for secondary text, borders, disabled states

4. **Component Selection Rules** — When to use which existing component or shadcn primitive. Decision tree format:
   - Need a status indicator? -> Badge.tsx (not a colored div)
   - Need a data container? -> shadcn Card (not a plain div with classes)
   - Need a CTA button? -> shadcn Button with accent variant
   - Need to show agent reasoning? -> AgentDecisionPanel pattern

5. **Layout Rules** — Max width constraints, padding standards, responsive approach (extracted from actual components).

6. **Copy/Language Guide** — Preferred terminology table: use "listing" not "ticket", "lock funds" not "deposit", "settle" not "finalize", "verdict" not "classification result". Extract actual terms used across components.

7. **Anti-patterns** — Things to never do:
   - Never use light backgrounds
   - Never show raw contract addresses without EtherscanLink
   - Never hide the agent's confidence score
   - Never use "blockchain" or "smart contract" in user-facing copy
   - Never add features that take more than 5 seconds to understand in a demo

8. **Reference** — Point to STYLE-GUIDE.md for exact tokens, index.css for CSS variables, and the component files for implementation examples.

Write this as a prompt-ready document — it should work when pasted into a Claude context window with zero modification.
